Spring缺点

第一个是配置繁琐

第二个是依赖繁琐

SpringBoot功能

第一个是自动配置，SpringBoot的自动配置就是运行时，开率了众多因素，决定了SpringBoot使用哪一个，不适用哪一个。

第二个起步依赖，起步依赖默认是Maven项目对象模型，定义了对其他库的传递依赖。

第三个是辅助功能，提高大型项目安全性

Spring是为了解决企业级应用开发的复杂性而创建的

微服务

MVC三层架构，MVVM 微服务架构

微服务架构优化了以上问题，但是也带来新的问题，这样给应用开发带来了简便，但是却给运维带来了难题。

自动装配的原理

  • 首先是pom.xml，

    • spring-boot-dependenciex：核心依赖在父工程中
    • 然后是父依赖里面有对依赖的版本管理和yaml文件
    • 引入依赖的时候不需要指定版本，因为使用的是父类写好的

  • 启动器，作用就是自动导入相关的依赖

        <dependency>
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-starter-web</artifactId>
        </dependency>
    • spring-boot把这些使用场景作为了启动器，需要什么东西，直接启动

  • 注解@SpringBootApplication是一个组合注解

    • @SpringBootConfiguration:（核心）

      这个注解是 @Configuration 的一个特化版本，表示该类是一个配置类，可以用来定义 Spring 容器中的 bean。

      • Configuration：Spring配置类，在spring里面学习的一个配置类
        • Component：一个组件

    • @EnableAutoConfiguration:（核心）

      这个注解告诉 Spring Boot 开始自动配置你的应用程序。这意味着 Spring Boot 会根据你应用中添加的依赖自动配置 Spring 框架。例如，如果类路径中有 spring-boot-starter-web 依赖，那么 Spring Boot 会自动配置一个 web 服务器和 Spring MVC。

      • AutoConfigurationPackage：自动配置包

        • @Import({AutoConfigurationPackages.Registrar.class})
public @interface AutoConfigurationPackage {
    String[] basePackages() default {};

    Class<?>[] basePackageClasses() default {};
}

          导入了一个选择器，进一步的看，这里的注解的Registrar通过导入元数据来注册了一些配置。

      • @Import({AutoConfigurationImportSelector.class})：自动导入选择

        • 进一步查看这个类，他有方法会获取pom里面选择的启动器，这是一个加载元数据的过程，还可以获取环境

        • 然后还有获取配置的过程：核心

          protected AutoConfigurationEntry getAutoConfigurationEntry(AnnotationMetadata annotationMetadata) {
        if (!this.isEnabled(annotationMetadata)) {
            return EMPTY_ENTRY;
        } else {
            AnnotationAttributes attributes = this.getAttributes(annotationMetadata);
            List<String> configurations = this.getCandidateConfigurations(annotationMetadata, attributes);
            configurations = this.removeDuplicates(configurations);
            Set<String> exclusions = this.getExclusions(annotationMetadata, attributes);
            this.checkExcludedClasses(configurations, exclusions);
            configurations.removeAll(exclusions);
            configurations = this.getConfigurationClassFilter().filter(configurations);
            this.fireAutoConfigurationImportEvents(configurations, exclusions);
            return new AutoConfigurationEntry(configurations, exclusions);
        }
    }

          核心是：

          List<String> configurations = this.getCandidateConfigurations(annotationMetadata, attributes);
//核心方法：
protected List<String> getCandidateConfigurations(AnnotationMetadata metadata, AnnotationAttributes attributes) {
        List<String> configurations = ImportCandidates.load(AutoConfiguration.class, this.getBeanClassLoader()).getCandidates();
        Assert.notEmpty(configurations, "No auto configuration classes found in META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports. If you are using a custom packaging, make sure that file is correct.");
        return configurations;
    }

    • @ComponentScan:

      这个注解启用了组件扫描（component scanning），Spring 会扫描该类所在包及其子包中的所有组件（包括 @Component、@Service、@Repository、@Controller 等注解标记的类），并将它们注册到 Spring 容器中。

依赖注入

一般我们写在配置文件里面。首先是依赖的前提：

第一个是需要一个组件注解（@Component），使之成为一个Springboot的组件，使得这个类可以被扫描到。

第二个是赋值需求，我们需要一个新的注解（@Vale(“value”)），然后

    @Value("小明")
    private String name;

    @Value("1")
    private int age;

然后在测试类里面:
    @Autowired
    Dog dog = new Dog();

但是新的方式：通过yaml来赋值。同时ConfigurationProperties的作用是把当前类和配置绑定。

自动装配原理

自动配置位置，能够自动装备的原因就是配置是从这里导入的，而这里面的全部都已经配置完毕了，但是由于没学过spring，所以这个配置我并不是很理解，之后再补

org:springframework.boot:spring-boot-autoconfiguer:
  spring-boot-test-autoconfiguer-version.jar
    META-INF
      spring.factories

其中所有的自动配置文件名的格式为xxxAutoConfiguration，自 Spring Boot 3.0 开始，自动配置包的路径从META-INF/spring.factories 修改为 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports。

自动装配文件的格式：

@AutoConfiguration(
    after = {JacksonAutoConfiguration.class}
)
@ConditionalOnClass({Cluster.class})
@Conditional({CouchbaseCondition.class})
@EnableConfigurationProperties({CouchbaseProperties.class})
public class CouchbaseAutoConfiguration {
    private final CouchbaseProperties properties;

    CouchbaseAutoConfiguration(CouchbaseProperties properties) {
        this.properties = properties;
    }
类定义就是注解spring的Configuration要求的，没了解过

要加载的类JacksonAutoConfiguration，@EnableConfigurationProperties加载器，加载指定的配置，（把类参数传入类加载器里面，或者说绑定在一起）另外的是配置文件和这个加载源的关系：

@ConfigurationProperties(
    prefix = "spring.couchbase"
)
public class CouchbaseProperties {
    private String connectionString;
    private String username;
    private String password;
    private final Env env = new Env();

此类的prefix的值spring.couchbase和配置文件里面的配置很像，或者说是此配置prefix = "spring.couchbase"直接和这个spring.couchbase绑定了。

直观的看就是，此自动配置和下面绑定：

spring:
  couchbase:
    属性

还有一个重要的spring底层注解，就是符合条件的配置/类就直接开启。

@ConditionalOnMissingBean

合起来就是：xxxAutoConfiguration和xxxProperties绑定在一起（默认值，由EnableConfigurationProperties去加载xxxProperties），进一步的和配置文件绑定，用于更改默认值。
